## skills/skill-creator/SKILL.md
**Type**: Meta-skill / workflow orchestration — full lifecycle for creating, iterating, evaluating, and optimizing Claude skills.

**Portable**: Partially. The conceptual SOP (draft → eval → review → iterate → optimize description) is universally portable and strong. The implementation is tightly coupled to bundled Python scripts (`eval-viewer/generate_review.py`, `scripts/aggregate_benchmark.py`, `scripts/run_loop.py`, `scripts/package_skill.py`) and subagent reference files (`agents/grader.md`, `agents/comparator.md`, `agents/analyzer.md`) that are Anthropic-internal and not transferable as-is.

**Reason**: The high-level loop and its guiding principles (generalize from feedback, keep prompts lean, explain the why, avoid overfitting to examples, bundle repeated helper scripts) are platform-agnostic and directly applicable to any skill-creation workflow. The eval architecture (parallel with-skill + baseline runs, assertions, benchmark aggregation, browser viewer) requires porting or replacing tooling.

**Trigger**: User wants to create a skill from scratch, improve or optimize an existing skill, run evals to test a skill, benchmark skill performance with variance analysis, or optimize a skill's description for better triggering accuracy.

**Steps/contract**:
1. **Capture Intent** — what the skill does, when it triggers, output format, whether test cases are warranted
2. **Interview & Research** — edge cases, input/output formats, dependencies, success criteria; research via MCPs in parallel
3. **Write SKILL.md** — name, description (primary trigger mechanism; lean "pushy"), compatibility, body; anatomy: metadata + body + bundled resources
4. **Write test cases** — 2–3 realistic user prompts; save to `evals/evals.json` (prompts only, no assertions yet)
5. **Spawn parallel eval runs** — with-skill AND baseline in the same turn; create `eval_metadata.json` per run; draft assertions while runs execute
6. **Grade → aggregate → analyst pass → launch viewer** — grade assertions, run aggregation script, surface non-discriminating/flaky/tradeoff patterns, present to user via `generate_review.py`
7. **Read feedback → improve skill** — generalize from feedback, keep prompt lean, explain why, bundle repeated patterns into `scripts/`
8. **Iterate** — rerun into `iteration-N+1/`, pass `--previous-workspace` to viewer; repeat until feedback is empty or no meaningful progress
9. **Description optimization** — generate 20 trigger eval queries (10 should/10 should-not, realistic and tricky near-misses), review with user via HTML eval reviewer, run `scripts/run_loop.py` (60/40 train/test split, up to 5 iterations, select by test score), apply `best_description`
10. **Package** — `scripts/package_skill.py` → `.skill` file; present to user

**Strip**:
- All bundled script invocations (`generate_review.py`, `aggregate_benchmark.py`, `run_loop.py`, `package_skill.py`, `eval_review.html`) — replace with adopter's own tooling or describe steps manually
- Timing data capture via subagent notification (platform-specific artifact)
- Cowork-specific and Claude.ai-specific adaptation sections (too narrow)
- Blind comparison section (optional advanced feature, subagent-dependent)
- `agents/grader.md`, `agents/comparator.md`, `agents/analyzer.md` references (bundled, not portable)
- "billions in economic value" motivational aside (internal flavour)

**Structure/format**: ~500+ lines, well-organized with clear phase headers and generous code blocks (JSON schemas, bash commands, example patterns). Writing style guidance is notably high quality. Has a good progressive disclosure anatomy diagram. Could be distilled to ~150-line SOP by keeping the loop structure, principles, eval design guidance, and description-optimization rationale while dropping script-specific mechanics.

**Notes**:
- The **writing style section** is the most portable gem: prefer explaining *why* over MUSTs; use theory of mind; generalize from examples rather than overfit; remove instructions that aren't pulling weight; bundle repeated work into scripts. These principles should be extracted verbatim or near-verbatim.
- The **description optimization guidance** (realistic concrete queries with personal context, near-miss negatives, train/test split to avoid overfitting) is strong and reusable independently of the optimization script.
- The **progressive disclosure architecture** (metadata ~100w always in context, SKILL.md body <500 lines, bundled resources as needed) is a clean structural rule worth adopting.
