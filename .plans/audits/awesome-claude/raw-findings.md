## break-stop.md

**Type**: Rule — CI/check gate with hard-stop protocol and escalation ceremony
**Portable**: Yes
**Reason**: The break-stop pattern (run checks after every change, halt on failure, surface to user rather than self-fix) is universally applicable across any project that runs automated checks. No repo-specific tooling is hardcoded — `make check` is cited as an example and is clearly swappable.
**Trigger**: After every logical change; also pre-emptively when a requested change will predictably break existing tests, types, or contracts.

**Structure/format**: Well-structured: Rule → Protocol → Proactive Protection → Severity Levels → Output Format → Rationale. The proactive-warning section (warn before applying a predictably breaking change) is the strongest differentiator — most rules only address post-hoc breakage.
**Notes**: The "you do NOT have authority to fix broken functionality on your own" framing is a strong, transferable principle. Pairs naturally with a TDD or CI skill. The severity split (trivial auto-fix vs. hard stop) prevents alert fatigue, which is rare in rules of this type and worth preserving.

## TRANSACTIONS.md

**Type**: Architecture rule set — DDD transaction boundaries, outbox pattern, and event delivery contract for write-model operations
**Portable**: Partial
**Reason**: The core transaction discipline (one aggregate per transaction, no network calls inside, optimistic locking, retry with backoff, outbox written in same transaction as aggregate) is universally sound DDD/CQRS guidance. The outbox schema contract (event_id, event_type, event_version, aggregate_id, status, retry_count, etc.) is portable. However the full document assumes a DDD aggregate model with CQRS read/write split — teams not using that architecture would need to strip or adapt roughly half the content.
**Trigger**: Code touches `src/infrastructure/db/**` or `migrations/**`; or when writing application services / command handlers that persist domain aggregates; or when setting up event publishing from a write model.
**Steps/contract**: (1) Open transaction at application-service / command-handler level only. (2) One aggregate, one repository, one transaction — never span multiple aggregates. (3) No network calls, queues, HTTP, or sleeps inside a transaction. (4) Choose isolation level explicitly (READ COMMITTED default; REPEATABLE READ for stable snapshot; SERIALIZABLE only with justification). (5) Optimistic locking via `version` field; conflict → retry with backoff and attempt limit at the application-service level. (6) Write outbox record in the same transaction as the aggregate commit — if the aggregate is committed the event must exist.
