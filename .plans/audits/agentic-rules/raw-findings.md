**Type**: sop
**Portable**: partial
**Reason**: Covers eight broadly useful SOPs (backlog, planning, CoT, TDD, refactoring, CI/CD, rollback/monitoring, self-validation) but is tightly coupled to AINative Studio / MCP-as-a-bus and AWS HyperScaler tooling throughout.
**Trigger**: Use when an agent needs a comprehensive operating ruleset covering the full SDLC—from issue triage through deployment and monitoring—within a GitHub + MCP + serverless cloud stack.
**Steps/contract**:
- §1 Backlog: fetch highest-priority GitHub issue → classify/label → branch as `feature/{issue-number}-{slug}` → post WIP comment via MCP → TDD loop → PR against `develop` → run checks → merge → close issue
- §2 Planning: emit 4-week MVP roadmap table → dependency DOT graph → release checklist → risk assessment (≥3 risks with Likelihood + mitigation)
- §3 CoT gate: before nontrivial code emit numbered narrative — (1) clarify, (2) weigh alternatives, (3) edge cases, (4) chosen plan
- §4 TDD: Red (failing test) → Green (minimal impl) → Refactor; coverage ≥ 90%; test dirs `/unit`, `/integration`, `/functional`
- §5 Refactoring: extract helpers, callback→async/await, guard clauses, strip unused imports
- §6 CI/CD: MVP pipeline YAML (checkout → install → test → deploy → smoke), Terraform snippet, canary deploy (5-hit health check → 25% traffic shift or revert)
- §7 Rollback: read `stable_version.json` → `aws lambda update-alias` → MCP message; monitor every 60 s → POST alert webhook on failure; metrics dashboard with `jq` error-rate formula
- §8 Self-validation: after each task — (1) format correct? (2) reasoning present? (3) names/placeholders correct? — if blocked, state rule, reason, and missing info
**Strip**: All AINative Studio / IDE-specific references; MCP-as-notification-bus (Slack/Teams via MCP); AWS Lambda / HyperScaler-specific Terraform and CLI commands; HyperScaler secret placeholder names; `rollback.sh` and `monitor.py` implementation details; `metrics-dashboard.md` AWS CloudWatch specifics; branch-protection-via-MCP-notification details; "Merged 2026-03-27" provenance header
**Structure/format**: Single flat Markdown file, §1–§8 numbered sections; §1–§7 are prose+table+code-block, §8 is a compact three-question audit + fail-safe declaration. No YAML frontmatter. Sections are self-contained and individually extractable. §8 (self-validation) is the most reusable in isolation and matches the existing `self-validation` skill pattern.
**Notes**: §3 (CoT gate) and §8 (self-validation) are already strong standalone skills in this repo (`cot-gate`, `self-validation`). §1 (backlog/GitHub Issues) and §4 (TDD) partially overlap `issue-triage` and `tdd` skills.
