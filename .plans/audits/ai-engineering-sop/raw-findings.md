## ai/skill/plan-to-spec.md
**Type**: Skill definition — workflow for converting a plan or phase slice into narrow, reviewable task specs (the plan→spec bridge)
**Portable**: Yes
**Reason**: Entirely tool-neutral and stack-neutral. No project-specific logic; the nine-step workflow (slice → one-outcome → split → execution contract → status/checklist → validation → white-box judgment → write-back → failure boundaries) applies to any agent-assisted or human engineering workflow.
**Trigger**: When a plan or phase slice is available and work is about to move into implementation — i.e., whenever the bridge from "we have a plan" to "we have actionable specs" needs to be crossed.
**Steps/contract**: (1) Start from the current slice — narrow to the smallest reviewable unit that still advances the phase. (2) One spec, one primary outcome — split if mixing concerns (design + impl + migration + cleanup). (3) Split before the spec becomes large (multiple outcomes, independent validation paths, checklist that can't stay short). (4) Make the execution contract explicit: in-scope, out-of-scope, affected area, done condition. (5) Add lightweight execution state: status enum (`draft`→`done`) + short Markdown checklist; checklist completion alone is not done. (6) Make validation explicit — black-box checks as the default acceptance path. (7) Judge white-box need: trigger when logic is branch-heavy, stateful, regression-sensitive, contract-sensitive, or tied to a deterministic bugfix. (8) Decide write-back deliberately: only when the task is expected to clarify stable, reusable context; name the destination. (9) Add failure boundaries (Repair Budget, Rollback Scope, Escalation Condition) only for long-running or dependency-heavy work; omit on trivially narrow tasks.
**Strip**: `ai/doc/specs/` path references and `ai/doc/specs/README.md` convention (repo-specific directory layout). Replace with generic "task spec output directory (per-project convention)". Phase-aware field names (`project_target`, `current_target`, `main_plan`/`sub_plan`) are opinionated to this repo's hierarchy label — keep as illustrative examples, not mandatory schema.
**Structure/format**: H2-section markdown with a clean nine-step numbered workflow plus a "Common failure modes" anti-pattern checklist. Both sections are self-contained and convert directly to a skill file with no restructuring. The failure-modes list is independently portable as a self-audit block.
**Notes**: The strongest standalone skill in this repo. The "Common failure modes" section (12 named anti-patterns) is exceptionally high-value and not replicated elsewhere — worth preserving verbatim. The failure-boundary fields (`Repair Budget`, `Rollback Scope`, `Escalation Condition`) are absent from the existing target-repo `plan-to-spec` skill and represent a clear upgrade. ## ai/skill/design-whitebox-tests.md
**Type**: Skill definition — decision procedure for determining whether white-box validation is needed and identifying which internals to protect
**Portable**: Yes
**Reason**: Entirely tool-neutral and stack-neutral. No project-specific paths, naming conventions, or tooling. The trigger conditions (branch-heavy logic, state transitions, deterministic bugfixes, caches/parsers/schedulers), protect/avoid guidance, and failure modes apply universally to any codebase or agent validation workflow.
**Trigger**: When a task changes branch-heavy logic, internal state transitions, or fixes a deterministic bug — i.e., at the white-box/black-box decision point described in `validation.mdc` and `AGENTS.md`. Complements rather than replaces black-box acceptance.

(5) Output: yes/no on white-box need + what to protect + what to skip + suggested regression protection direction.
