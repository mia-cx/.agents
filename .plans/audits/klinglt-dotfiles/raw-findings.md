**Strip**: Hardcoded `npm run build` / `pnpm build` references — replace with a `<build command>` placeholder; the title's implicit TypeScript framing (the procedure itself is already language-agnostic)

## mcp-configs/mcp-servers.json
**Type**: Configuration reference / MCP server catalog template — JSON file listing 15 MCP server definitions across two transport shapes (stdio via `npx` and HTTP via `url`), with a `_comments` meta-block carrying usage instructions
**Portable**: Partially — the dual-transport schema (stdio: `command`/`args`/`env` vs. HTTP: `type: "http"` / `url`), placeholder conventions (`YOUR_*_HERE`), and the `_comments` rules are portable; the curated server list (GitHub, Firecrawl, Supabase, Vercel, Railway, Cloudflare, ClickHouse, etc.) is illustrative rather than project-specific but is ultimately a personal selection, not a universal canon
**Reason**: The file's portable value lies in two things: (1) the JSON schema it demonstrates — a self-documenting catalog format where each entry carries a `description` field and the whole file has a `_comments` governance block; (2) two concrete, enforceable setup rules buried in `_comments`: keep active MCPs under 10 to preserve context window, and use `disabledMcpServers` in project config for per-project scoping. The server list itself has no project-specific entries and could serve as a starter menu, but individual server choices are personal and ecosystem-dependent.
**Trigger**: When configuring Claude Code (or any MCP-aware client) with a new set of servers — onboarding a machine, setting up a project workspace, or auditing context-window pressure from too many active MCPs
**Steps/contract**: (1) Identify the servers needed for the target workflow; (2) Copy matching entries from the catalog into `~/.claude.json` under `mcpServers`; (3) Replace all `YOUR_*_HERE` env-var placeholders with real credentials; (4) For project-scoped exclusions, add server keys to `disabledMcpServers` in the project config; (5) Verify total active server count stays below 10 to protect the context window
**Strip**: The specific server entries are illustrative — a portable SOP should document the schema and rules, not prescribe a fixed server list; `--project-ref=YOUR_PROJECT_REF` in the Supabase `args` array is a useful pattern illustration worth keeping as a schema example; the `supabase`, `firecrawl`, `magic`, `railway`, and `clickhouse` entries are niche enough that a generic catalog would make them optional examples
**Structure/format**: Well-formed JSON with inline `description` per entry and a top-level `_comments` governance object — an unusual but self-documenting format for configuration reference; the `_comments` block is the closest thing to a procedural SOP and could be extracted verbatim as a setup checklist
**Notes**: The two rules in `_comments` are the highest-value portable artefacts: (a) the 10-MCP context-window cap is a specific, measurable, and immediately enforceable constraint rarely documented elsewhere; (b) the `disabledMcpServers` pattern for per-project scoping is a useful operational discipline. The dual-transport schema (stdio vs. HTTP) is worth preserving as a reference in any MCP setup skill. Low standalone promotion priority — best absorbed into an existing `mcp-builder` or setup-mcp skill as a "starter catalog + governance rules" appendix rather than a standalone SOP. The context-window cap rule is the single artefact most worth carrying forward verbatim.

## contexts/review.md
**Type**: Context snippet / lightweight behavioral brief — plain markdown, no YAML frontmatter, no subagent wrapper
**Portable**: Partially — the severity-priority ordering (critical > high > medium > low), the "suggest fixes, don't just point out problems" behavioral rule, and the 7-item checklist categories are stack-agnostic and portable; the file as a whole is too thin to stand alone as a skill
**Reason**: The checklist covers the canonical code-review surface (logic, edge cases, error handling, security, performance, readability, test coverage) with no language or project coupling. The two behavioral directives — read thoroughly before commenting, and suggest fixes rather than just flag issues — are universally applicable review disciplines. However, the file provides no severity taxonomy definitions, no per-issue output template, no approval-verdict mapping, and no structured report format; it is a behavioral brief, not a procedural SOP.
**Trigger**: Before reviewing any code change or PR; implied by "Mode: PR review, code analysis" header — fires at review time
**Steps/contract**: (1) Read code thoroughly before commenting; (2) Prioritise findings by severity (critical > high > medium > low); (3) For each finding, suggest a concrete fix rather than only identifying the problem; (4) Check security vulnerabilities explicitly; (5) Apply 7-category checklist (logic errors / edge cases / error handling / security / performance / readability / test coverage); (6) Group output findings by file, severity first
**Strip**: Nothing needs stripping — the file is already minimal; the "Mode:" and "Focus:" header labels are presentational scaffolding that can be collapsed into a skill description
**Structure/format**: Very terse — a two-line header block, a 4-item behavioral list, a 7-item markdown checklist, and a single-sentence output-format directive. No per-issue output template, no severity definitions, no verdict system, no fenced code examples. The checklist is the only portable formatting artefact.
**Notes**: Substantially weaker than `agents/code-reviewer.md` already catalogued in this repo, which covers the same domain with explicit severity-tier definitions (CRITICAL/HIGH/MEDIUM), a per-issue output template (severity / file:line / issue / fix / bad→good snippet), and a three-state approval verdict (✅/⚠️/❌). This file adds nothing not already present in that agent definition and is missing all structural scaffolding that makes a review SOP actionable. Low standalone promotion priority. The one marginal addition worth folding into a unified review skill is the explicit "suggest fixes, don't just point out problems" behavioral directive as a named output contract — it is concise, enforceable, and absent from many review prompts.

## examples/CLAUDE.md
**Type**: Example/template project-level CLAUDE.md — a skeleton configuration document providing declarative coding standards, not a procedural workflow SOP
**Portable**: Partially — the coding-standards rules (immutability, no console.log, error handling, Zod validation), security rules (no hardcoded secrets, env vars, parameterised queries, CSRF), TDD discipline (write tests first, 80% coverage, test-type taxonomy), and Git workflow rules (conventional commits, no direct-to-main, tests gate merge) are all portable; the file-structure section, TypeScript API-response interface, and environment-variable block are Next.js/TS-specific scaffolding
**Reason**: This is a template meant to be copied and customised per project. Its individual rules are sound and reusable as declarative constraints, but the file is not a workflow SOP — it has no phases, no agent steps, no procedural loop. Most of the substantive rules overlap with more detailed SOPs already audited from this repo (TDD → `agents/tdd-guide.md`; security → `agents/security-reviewer.md`; code review → `agents/code-reviewer.md`). The marginal portable contributions are the file-size guidance (200–400 lines typical, 800 max), the "organize by feature/domain, not by type" architecture rule, and the explicit "all tests must pass before merge" gate.
**Trigger**: Project initialisation / onboarding — when setting up a new project's CLAUDE.md or standing rules document; not triggered by a runtime event or user command
**Steps/contract**: Declarative constraints only — not a procedural SOP. Rules cluster into four categories: (1) Code organization: ≤800 lines/file (200–400 typical), feature/domain organization, high cohesion/low coupling; (2) Code style: no emojis, immutable patterns, no console.log in production, try/catch with user-friendly messages, Zod or equivalent for input validation; (3) Testing: write tests first, 80% minimum coverage, unit/integration/E2E tiers; (4) Security: no hardcoded secrets, env vars for sensitive data, validate all inputs, parameterised queries only, CSRF protection; (5) Git: conventional commits (`feat:` / `fix:` / `refactor:` / `docs:` / `test:`), no direct commits to main, PR review required, all tests must pass before merge
**Strip**: File structure section (Next.js app-router tree — project-specific); `ApiResponse<T>` TypeScript interface and error-handling fenced code blocks (illustrative, stack-specific); Environment Variables block (project-specific placeholders); Available Commands list (`/tdd`, `/plan`, `/code-review`, `/build-fix` — host-specific slash commands); Project Overview placeholder paragraph
**Structure/format**: H2 sections with bullet lists and fenced code blocks — configuration/reference document style, not an executable workflow. No phases, no output templates, no agent steps. Works better as a standing rules reference to inject into a project context than as a skill with a trigger and steps. Lowest structural density of any file audited in this repo.
**Notes**: Low standalone SOP promotion priority. This file's primary value is as a starting-point template for project-level CLAUDE.md files, not as a portable workflow SOP. The three portable rules worth extracting into existing skills are: (a) the file-size ceiling (800 lines hard max, 200–400 typical) — concrete and immediately enforceable, absent from most other files audited; (b) "organize by feature/domain, not by type" — a concrete, opinionated architecture decision rule; (c) "all tests must pass before merge" as an explicit blocking gate. All other rules are better-developed in the specialist agent SOPs already catalogued. Do not promote as a standalone skill; instead, fold the three unique rules into a project-setup or style-guide rules artefact.
