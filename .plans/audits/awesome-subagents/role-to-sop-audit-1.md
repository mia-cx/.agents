# Role-to-SOP Audit — awesome-subagents (awesome-claude-code-subagents)
**Auditor:** worker agent  
**Source:** `.plans/audits/awesome-subagents/raw-findings.md`  
**Date:** 2026-03-29

---

## 1. Repo Overview

The `awesome-claude-code-subagents` repo is a community-maintained catalogue of ~100 Claude Code subagent definitions spanning ten domain categories (core development, language specialists, infrastructure, quality/security, data-AI, developer experience, specialized domains, business/product, meta-orchestration, and research analysis) plus a `.references/awesome-subagents/tools/subagent-catalog/` slash-command suite. Each file is a system-prompt persona defining a role specialist — not a procedural runbook. The corpus is heavily templated: nearly every agent follows an identical structural skeleton (YAML frontmatter → persona intro → SLO checklist → taxonomy bullet sections → JSON "Communication Protocol" → 3-phase workflow → JSON progress snippets → "Integration with other agents" list). This templating creates a strong signal-to-noise problem: the common scaffold is non-portable boilerplate, while the portable value sits inside the domain-specific checklist and SLO content buried within it. A pervasive anti-pattern across the repo is the injection of hardcoded fake metrics into delivery-notification templates ("23 vulnerabilities fixed", "47 files reviewed", "87% compliance score") — these undermine credibility and must be stripped before any SOP promotion. The JSON "Communication Protocol" blocks reference a fictional "context manager" agent that does not exist outside this repo's multi-agent ecosystem; they carry zero portable value. Language-specialist files (categories 02) account for roughly 23 of the ~105 files and are predominantly not portable — they are language-locked reference cards, not cross-domain SOPs. The highest-density portable SOP material concentrates in categories 01 (core development), 04 (quality/security), and 06 (developer experience). After stripping the boilerplate scaffold, approximately 30–35 files yield portable SOP kernels; roughly 45 are non-portable or too language-specific to generalise.
