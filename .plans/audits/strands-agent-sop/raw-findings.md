
## rules/agent-sop-format.md
**Type**: Meta-SOP / format specification — defines the canonical structure for `.sop.md` files in this repo (sections, naming, RFC2119 keywords, negative-constraint framing).
**Portable**: Partially — the structural contract (Overview / Parameters / Steps / Examples / Troubleshooting) and RFC2119 convention are universally applicable; the `.sop.md` file-extension convention and Cursor rule wrapper (`<rule>` block / `globs`) are repo-specific and must be stripped.
**Reason**: Pure format governance with no domain logic. Any agent ecosystem that authors SOPs would benefit from this exact template and the negative-constraint-with-context rule. The RFC2119 keyword definitions and the "provide context for every MUST NOT" mandate are high-signal and rarely stated this explicitly elsewhere.
**Trigger**: Activate when creating a new SOP, auditing an existing SOP for format compliance, or onboarding contributors who will write SOPs.
**Steps/contract**: (1) Name files in kebab-case; (2) include five canonical sections (Title+Overview, Parameters, Steps, Examples, Troubleshooting); (3) snake_case parameter names; (4) RFC2119 keywords for all constraints; (5) every negative constraint must supply a `because/since/as` rationale clause; (6) ask all required parameters in one prompt; (7) save artifacts to specified paths.
**Strip**: `<rule>` / `</rule>` XML block, `globs`, `filters`, `actions`, `metadata` front-matter fields, and the Cursor-specific "suggest" action wrapper. The examples inside the rule block duplicate content already in the spec — collapse to one canonical copy.
**Structure/format**: Well-structured with code-fenced templates for each section. The negative-constraint examples (good vs. bad) are an excellent teaching pattern worth preserving verbatim. Minor issue: list item 6 is numbered 7 in the original (typo — step 6 is missing).
**Notes**: This is the strongest format-governance artefact in the repo. The "negative constraints require context" rule is a standout portable principle — directly addresses a common LLM prompt failure mode (seeding the model to consider the forbidden action). Consider promoting to a cross-cutting rule rather than burying it inside a format spec.

**Reason**: Most complete code-implementation SOP in this repo. The documentation/code separation principle (docs describe, code lives in repo_root) is a clean architectural constraint worth carrying forward verbatim.
**Trigger**: Activate when implementing a feature, bug fix, or utility function using TDD in any codebase.
**Steps/contract**: (1) Setup — validate/create documentation directory, discover instruction files (e.g. CODEASSIST.md), create context.md and progress.md; (2) Explore — analyse requirements, research existing patterns and dependency map, compile code context document; (3) Plan — design test strategy with explicit input/output pairs, create implementation plan saved to plan.md; (4) Code — implement all tests first (RED), implement code to pass tests (GREEN), refactor to align with codebase conventions, validate all tests and builds pass; (5) Commit — conventional commit message, stage all relevant files, no push.
**Strip**: `CODEASSIST.md` integration is strands-specific — generalise to "project-specific instruction file at repo root" or omit. Default `documentation_dir` of `.agents/scratchpad/{project_name}` is strands-specific — use a generic placeholder. The "🤖 Assisted by the code-assist SOP" commit footer is repo-branding, drop. The specific `find` command in Step 1 for instruction-file discovery is an implementation detail, not a portable constraint — replace with "search repo root for instruction/convention files". Multi-package coordination troubleshooting block is too specific to monorepo setups. Step references inside constraints are misaligned (Step 4.4 says "return to step 5.2" — an internal numbering error to fix).
**Structure/format**: Excellent. The mode-bifurcation pattern (one `## Mode Behavior` section, every step ends with a `> 💬 See Mode Behavior` callout) is a clean, reusable template device. Desired Outcome section doubles as acceptance criteria — keep. Best Practices section (project detection, build output to log files, consolidated doc files) is a high-value appendix. Length is large but every section carries weight; unlike other SOPs, the Troubleshooting block is genuinely actionable rather than boilerplate.
**Notes**: The interactive/auto mode bifurcation is the standout transferable innovation — defining both modes once and referencing throughout avoids duplication and keeps each step clean. The "implement ALL tests before ANY implementation code" and "only seek user input when stuck" (in auto mode) constraints are stronger than generic TDD guidance and should be extracted as portable rules. The `build output to log file, grep for pass/fail` pattern (Build Output Management) is a practical reliability constraint worth promoting as a standalone best practice.

**Notes**: The one-question-at-a-time clarification loop (step 4) is the single highest-value portable pattern here — it prevents user overload and produces a traceable Q+A file for downstream steps.
